Во время запуска программы происходит последовательная инициализация компонентов и переход в интерактивный режим. Ниже представлен детальный процесс выполнения с привязкой к требованиям вашей лабораторной работы.

Порядок выполнения программы (Runtime Flow)
Проверка аргумента командной строки (Main.java:13)

Программа проверяет, передано ли имя файла при запуске.
Деталь для лабы: Согласно to do.md (п. 14), имя файла обязательно должно передаваться как аргумент командной строки.
Инициализация FileManager (Main.java:25)

Создается объект для управления файловыми операциями.
Деталь для лабы: Файловый менеджер использует формат JSON (п. 15), а для чтения и записи применяются FileReader и FileOutputStream (п. 16, 17).
Инициализация WorkerManager (Main.java:26)

Создается основной менеджер коллекции, в который передается FileManager.
Деталь для лабы: Внутри используется именно PriorityQueue (п. 12), а объекты Worker поддерживают автоматическую сортировку (п. 10).
Автоматическое заполнение коллекции (Main.java:29)

Вызывается fileManager.readCollection().
Деталь для лабы: Это критическое требование (п. 13) — коллекция должна заполняться данными из файла сразу при старте. Если файл пуст или отсутствует, создается пустая коллекция.
Создание менеджеров ввода (Main.java:31-33)

Инициализируются ConsoleInputManager (чтение из консоли), WorkerReader (сложная логика ввода полей рабочего) и ScriptReader (для команды execute_script).
Деталь для лабы: WorkerReader должен реализовывать ввод «по одному полю в строку» с приглашением к вводу и валидацией (п. 42-45).
Регистрация команд в CommandManager (Main.java:35)

Создаются экземпляры всех команд (от help до filter_by_salary). Каждой команде передаются необходимые зависимости (WorkerManager, WorkerReader и др.).
Деталь для лабы: Все реализованные команды должны соответствовать списку из задания (п. 22-37).
Запуск интерактивного режима (Main.java:37)

Вызывается commandManager.interactiveMode().
Программа входит в бесконечный цикл while(true), ожидая ввода пользователя.
Деталь для лабы: Программа должна корректно обрабатывать «неправильные данные» (п. 19), не «падая» при ошибках ввода, а предлагая повторить попытку (п. 45).
Важные уточнения для защиты (из to do.md):
IdManager / Автогенерация: Поля id и creationDate не должны вводиться пользователем (п. 47). В вашем коде это реализовано через IdManager.generateId().
Enums: При вводе полей Position, Status или OrganizationType программа должна сначала вывести список доступных констант (п. 44).
Null-значения: Если поле может быть null (например, salary или status), пользователь должен иметь возможность нажать Enter (пустая строка) для записи null (п. 46).
Javadoc: Перед защитой убедитесь, что все классы задокументированы через /** ... */ (п. 18).
